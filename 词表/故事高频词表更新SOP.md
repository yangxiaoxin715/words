# 故事高频词表更新 SOP

## 目标

用已经写好的英文故事，反推下一段词库中最值得保留和新增的词。词表服务故事理解，不按“更难”“更像教材”来选。

## 输入

- 已定稿或接近定稿的英文故事。
- 当前 `words-data.js`。
- 候选替换报告，建议放在 `词表/` 目录。

## 选词规则

1. 先统计故事中的词频和覆盖文件数。
2. 优先选影响理解的词：
   - 系列设定词，如 `compass`、`magic`。
   - 线索词，如 `ink`、`appear`、`page`、`line`。
   - 叙事动作和状态词，如 `stand`、`hold`、`fall`、`quiet`。
3. 人名、地名、历史专名、角色称呼不进词表。
4. 单篇故事里的情节词暂不进词表，除非它会成为系列核心词。
5. 不一次性重洗后 200。故事样本少于 30 篇时，建议每轮替换 30-50 个。

## 更新规则

1. 先保留当前更新段中已经被故事高频使用的词。
2. 优先替换当前故事样本中几乎不用、且更像教材主题词的词。
3. 新增词来源标为 `故事高频`。
4. 每一段保持 200 词；全词表总数按已上线段数计算，例如 3 段就是 600 词，目标 10 段就是 2000 词。
5. 事实性来源只维护在 `words-data.js`，测试只读取来源分布，不复制词表。

## 401-600 第三关规则

401-600 只给已经掌握前 400 词的孩子使用，不作为基础诊断入口。

建议结构：

1. 故事动作/状态：约 80 个。
2. 真实故事词形：约 50 个，重点补 `said`、`saw`、`went` 这类孩子在故事里直接看到的过去式。
3. 逻辑/线索词：约 40 个，用来支撑因果、转折、悬念和空间关系。
4. 情绪/人物变化：约 30 个，用来支撑人物成长线。

401-600 是新增第三关，不替换 201-400。上线时要同步修改词库模型、切组入口、完成 201-400 后的自动跳转和测试。

## 601-800 第四关规则

601-800 给已经掌握前 600 词的孩子使用，定位为“故事理解扩展词库”。这一段的重点是帮助孩子看懂故事里的细节、人物关系、原因结果和说明信息。

建议结构：

1. 故事场景和物件词：约 50 个。
2. 人物关系和互动词：约 35 个。
3. 细节描写词：约 35 个。
4. 阅读理解逻辑词：约 45 个。
5. 非虚构桥接词：约 35 个。

这一段先不要做年级分层。年级分层适合等 1000 词主干稳定后再做标签系统，否则后续每次扩词都要同时改主线和年级入口。

正式上线 601-800 时，不能只追加 `words-data.js`。页面分组入口也要改成动态生成，为 801-2000 留出结构。

## 801-1000 第五关规则

801-1000 给已经掌握前 800 词的孩子使用，定位为“主线阅读升级词库”。这一段继续服务历史罗盘故事主线，同时让孩子开始适应更长故事里的身份、动作、心理和阅读表达。

建议结构：

1. 主线历史和人物词：约 45 个。
2. 情节推进动作词：约 45 个。
3. 人物心理和态度词：约 35 个。
4. 语言文字和阅读表达词：约 35 个。
5. 长故事场景细节词：约 40 个。

这一段不能只按故事词频排序。当前故事样本里真正跨多篇出现、且未入库的词不多；如果只按词频，会被单篇故事里的情节词带偏。选词时要用故事证据做底盘，再补后续主线会反复用到的常用词。

正式上线 801-1000 时，要同时检查：

1. `words-data.js` 总数为 1000。
2. `WORD_POOLS` 自动生成 5 组，每组 200。
3. 首页自动出现 `801—1000`。
4. 完成 601-800 后能进入 801-1000。

## 1000词标签规则

1000 词主干稳定后，每个词都要自动带 3 类标签：

1. 段位，例如 `1—200`、`801—1000`。
2. 使用阶段，例如 `低年级可闪`、`高年级阅读升级`。
3. 故事作用，例如 `基础识别`、`阅读逻辑`、`历史人物`。

标签由 `words-data.js` 中的 `WORD_TAG_SCHEMA` 统一生成。不要在测试、文档或页面里复制 1000 个词的标签。

掌握地图导出的 CSV 要包含 `段位`、`使用阶段`、`故事作用`，方便后面做年级入口和复盘分析。

## 1001-1200 第六关规则

1001-1200 给已经掌握前 1000 词的孩子使用，定位为“长阅读桥接词库”。这一段的重点是帮助孩子从看懂故事，过渡到看懂阅读题、说明文、人物传记、历史短文和更长的主线文本。

建议结构：

1. 阅读题指令和题干词：约 35 个。
2. 说明文、传记、历史文本词：约 45 个。
3. 因果转折和观点表达词：约 45 个。
4. 长句结构和抽象动作词：约 40 个。
5. 复述写作输出词：约 35 个。

这一段不能只加更难的词。优先选能帮助孩子判断原因、结果、观点、证据、顺序和复述的词。

## 验证

每次更新后必须检查：

```bash
npm test
node -e "require('./words-data.js'); const words=globalThis.WORDS; const pools=Object.fromEntries(Object.entries(globalThis.WORD_POOLS).map(([k,v])=>[k,v.length])); console.log({total:words.length, unique:new Set(words.map(w=>w.english.toLowerCase()+'|'+w.chinese)).size, pools});"
```

通过标准：

- `npm test` 全部通过。
- `total` 等于当前已上线段数 × 200。
- `unique` 等于 `total`。
- `WORD_POOLS` 中每个上线分组长度都是 200。
- 每个词都有 `stageLabel`、`useStage`、`storyRole` 和 `tags`。

## 发布

如果线上页面由 GitHub Pages 提供，更新 `words-data.js` 后同步检查 `index.html` 的引用是否需要版本号：

```html
<script src="words-data.js?v=YYYYMMDD-topic"></script>
```

这样可以减少浏览器继续加载旧词表的概率。

本项目是纯静态页面，根目录保留 `.nojekyll`，让 GitHub Pages 直接发布文件，避免 Jekyll 处理词表文档。

Pages 发布产物只需要 `index.html`、`words-data.js` 和 `assets/`。`_config.yml` 会把 `tests`、`package.json`、`docs`、`词表` 排除出线上站点，词表文档仍保留在 GitHub 仓库中作为工作记录。

发布时必须确认真正上线分支。当前线上 Pages 使用 `main` 分支，不能只推功能分支。

发布检查：

1. `git ls-remote origin refs/heads/main` 确认 `main` 已到最新提交。
2. 检查 raw 文件，例如 `https://raw.githubusercontent.com/yangxiaoxin715/words/main/words-data.js`，确认词库总量正确。
3. 检查 Pages 带版本参数的地址，例如 `https://yangxiaoxin715.github.io/words/?v=YYYYMMDD-topic`，确认首页版本号和词库入口都更新。
4. 如果 GitHub Pages Actions 显示 build 成功但 deploy 失败，先确认 raw `main` 文件正确，再查看 deployment / jobs 状态；不要把 raw 成功当成 Pages 已上线。
5. 不带参数的 Pages 地址可能被 GitHub CDN 缓存约 10 分钟；如果带版本参数已经正确，说明发布已经生效，等待缓存过期即可。
