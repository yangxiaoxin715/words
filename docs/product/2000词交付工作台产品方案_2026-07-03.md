# 2000词交付工作台产品方案

## 需求定位

第一版网站要解决两个直接问题：

1. 用户使用更方便：闪卡、七天清单、查词查句、剧情猜想都在同一个网站完成。
2. 交付更轻松：你不用从微信里手工收集零散信息，后台可以直接用结构化数据生成下一集英文原文草稿。

微信后续主要用来了解孩子的学习状态、情绪、家庭执行情况。这类信息很难完全靠表单捕捉，适合继续通过聊天补充。

## 第一版核心闭环

第一版只跑通一条交付链路：

1. 孩子每天闪卡。
2. 家长在网站完成七天打卡清单。
3. 孩子在网站查关键句、查关键词，输入剧情猜想。
4. 后台汇总词汇数据、打卡数据、查句记录、剧情猜想和兴趣问题。
5. 你点击生成下一集英文原文草稿。
6. 你审核、微调，再进入音频生成和下一轮交付。

这条链路跑通后，再考虑更复杂的学情分析、家长看板和班级统计。

## 用户端功能

### 1. 闪卡数据

用户继续使用“单词猎人”闪卡。网站需要自动记录：

- 今天闪了多长时间。
- 今天闪了多少个单词。
- 新捕获多少词。
- 当前已捕获、瞄准中、潜伏中数量。
- 当前词段，例如 1-200、201-400、401-600。
- 最近一次闪卡时间。
- 哪些词反复瞄准中。

这部分数据回答一个问题：孩子的词汇接触和捕获进度到哪里了。

### 2. 七天打卡清单

把现在发给家长的 Day1-Day7 行动清单搬进网站。

用户每天打开当天页面，只做两件事：

- 按清单打勾。
- 填少量当天必须留下的信息。

每天页面不要展示太多解释，重点是让家长知道今天该做什么。

建议记录：

| 天数 | 用户要做的事 | 网站要收集的数据 |
|---|---|---|
| Day 1 | 裸听故事，记录听到的词和初步猜想 | 听到的词、猜的故事、孩子问题 |
| Day 2 | 看原文，圈认识词和看懂句 | 认识词、看懂句、想听出来的词 |
| Day 3 | 音文对齐，找卡住的地方 | 卡住句、播放速度、跟读感受 |
| Day 4 | 查关键句，回到音频重听 | 查过的句子、关键词、重听后是否听出 |
| Day 5 | 不看原文听大意，读 1-3 句 | 故事大意、读出来的句子 |
| Day 6 | 回答故事线问题，产生猜想 | 6 个问题答案、剧情猜想、猜想理由 |
| Day 7 | 回看 Day1，对比变化 | Day1 / Day7 对照、最终猜想、是否想看下一集 |

### 3. 站内查词查句

用户不用再跳到豆包或其他平台。

网站内提供一个受控查询工具：

- Day4 默认查关键句。
- 允许顺手查关键词。
- 每次查询都自动保存到孩子档案。
- 查询结果要适合家长带孩子理解，不写成大段翻译。
- Day1-Day6 不提供全文翻译按钮。
- Day7 结束后，可以提供全文对照，用来检查理解偏差。

查句结果建议固定成四块：

1. 这句话大概讲什么。
2. 哪几个词最关键。
3. 可以怎么根据上下文猜。
4. 回去重听时注意哪个声音或画面。

这部分数据回答一个问题：孩子卡在哪些句子，查完后有没有接回故事。

### 4. 剧情猜想和兴趣问题

孩子的剧情猜想要直接输入网站，不再散落在微信里。

需要收集：

- Day6 猜想。
- Day6 猜想理由。
- Day7 最终猜想。
- 孩子还想知道什么。
- 孩子最感兴趣的人物或问题。
- 孩子最喜欢的句子或画面。

Apple 和果冻的案例都说明，孩子的猜想常常会成为下一集故事的入口。后台生成下一集原文时，要把这些字段作为重要输入。

### 5. 今日提交

每天最后给用户一个“提交今日记录”按钮。

提交前只检查必要信息：

- 必选清单是否打勾。
- 当天关键输入是否填写。
- 如果用户点了全文翻译或输入大段翻译需求，系统提醒按当天流程走。

提交后用户看到简单反馈：

- 今日已完成。
- 明天继续第几天。
- 今天记录已进入孩子档案。

## 后台功能

### 1. 学员列表

后台首页先回答你每天最关心的几个问题：

- 谁今天完成了。
- 谁今天没完成。
- 谁卡在闪卡。
- 谁卡在故事句子。
- 谁已经到 Day6 / Day7，可以准备下一集。

列表字段：

- 学员昵称。
- 当前故事。
- 当前天数。
- 当前词段。
- 已捕获词数。
- 今日是否完成。
- 最近提交时间。
- 下一步动作。

### 2. 学员详情

学员详情页要把交付需要的信息放在一起：

- 基本信息：年级、教材、兴趣人物、感兴趣问题。
- 闪卡数据：总捕获、今日闪卡、当前词段、反复瞄准中的词。
- 七天清单：每天完成情况。
- 查句记录：查过哪些句子、关键词、重听结果。
- 剧情输入：Day6 猜想、Day7 最终猜想、孩子问题。
- 老师备注：你从微信聊天里补充的学习状态、情绪、家庭执行情况。

### 3. 下一集原文生成

后台提供“生成下一集原文草稿”按钮。

生成前先汇总这些输入：

- 当前故事标题和前情。
- 孩子兴趣人物或问题。
- Day6 / Day7 剧情猜想。
- 孩子喜欢的句子或画面。
- 当前已捕获词。
- 反复瞄准中的词。
- 本集查过的关键句和关键词。
- 需要复现的旧词。
- 目标新词数量。
- 单句长度限制。
- 预计总词数。

输出只做英文原文草稿，不自动生成音频，不自动发给用户。

输出格式：

```text
标题
英文故事正文
目标词表
旧词复现清单
需要人工检查的点
```

你审核后再决定是否使用。

### 4. 微信补充记录

网站收结构化数据，微信保留状态沟通。

后台需要有一个“老师备注”区，用来记录：

- 孩子今天情绪怎么样。
- 家长执行有没有跑偏。
- 孩子有没有抵触、兴奋、自豪、主动追问。
- 家里是否有特殊情况。

这些内容不强迫用户在网站填，避免打卡变重。

## 数据模型

第一版至少需要这些数据表或数据对象：

1. `learner`：孩子基本信息、年级、教材、兴趣人物、兴趣问题。
2. `word_state`：每个孩子每个词的状态，包含已捕获、瞄准中、潜伏中。
3. `flashcard_session`：每次闪卡的时间、词数、捕获变化。
4. `story_episode`：每集标题、正文、目标词、复现词、音频链接。
5. `day_checklist`：每一天有哪些清单项。
6. `day_submission`：孩子每天打勾和填写的结果。
7. `lookup_record`：查过的词、句子、AI 解释、重听结果。
8. `story_guess`：Day6 / Day7 的剧情猜想和理由。
9. `teacher_note`：微信沟通后补充的状态备注。
10. `generated_draft`：后台生成的下一集原文草稿和审核状态。

## MVP 优先级

### P0：第一版必须有

- 学员身份绑定。
- 闪卡数据同步。
- Day1-Day7 打卡清单。
- 每日提交。
- 站内查关键句和关键词。
- 剧情猜想输入。
- 学员详情页。
- 下一集英文原文草稿生成。

### P1：交付稳定后增加

- 家长端进度回看。
- 一键复制家长反馈。
- 卡住预警。
- 每 5 集词汇状态复测。
- 下一集目标词自动推荐。

### P2：规模扩大后再做

- 班级统计。
- 助教任务分配。
- AI 周报。
- 家长长期看板。
- 音频生成接入。

## 第一版暂缓

1. 复杂家长端 App。
2. 孩子自由 AI 聊天。
3. 排行榜。
4. 班级竞争。
5. 全自动故事发布。
6. 一上来接音频生成。
7. 复杂学情报告。

第一版先把“用户填写、数据归档、后台生成下一集草稿”跑通。

## 成功标准

第一版成功，看 7 个问题：

1. 用户是否能在网站完成闪卡和七天打卡。
2. 你是否能少收很多微信截图和零散文字。
3. 查词查句是否不用跳到外部平台。
4. Day6 / Day7 的剧情猜想是否能完整进入孩子档案。
5. 后台是否能看到一个孩子生成下一集所需的全部信息。
6. 下一集英文原文草稿是否能减少你的生产时间。
7. 微信聊天是否能更集中在学习状态和家长执行问题上。

如果这 7 个问题成立，再继续扩展后台分析和自动化能力。
